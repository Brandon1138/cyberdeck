# Codex App Server transport and durable worktree leases

## Evidence and compatibility boundary

A4 inspected only read-only installed metadata on 2026-07-21. `codex --version` reported
`codex-cli 0.144.6`. `codex app-server --help` documented `codex app-server --stdio` (stdio is also
the default transport), and the installed `generate-json-schema` metadata command produced official
v1 initialize plus v2 thread/turn schemas in a temporary directory. No App Server job was started
and no prompt was sent.

The observed protocol is newline-delimited RPC: requests declare JSON-RPC 2.0, while installed
Codex `0.144.6` responses and notifications omit the optional `jsonrpc` member:

1. request `initialize` with client identity;
2. validate the required initialize response and the `0.144.x` server user-agent family;
3. notify `initialized`;
4. request v2 `thread/start` with explicit cwd, sandbox, user-routed approvals, ephemeral storage,
   and a model only when the job supplied one;
5. request v2 `turn/start` with one text input;
6. correlate responses by JSON-RPC id and consume validated lifecycle notifications;
7. use `turn/interrupt` for cancellation or timeout.

The generated protocol has no negotiated numeric protocol-version field. Cyberdeck therefore
declares compatibility as the v2 method/schema family generated by Codex `0.144.x`, accepts the
observed omitted response-version member, rejects any explicitly incompatible JSON-RPC version, and
fails closed when initialize identifies another minor family or a required response shape is absent.
Upgrading Codex requires regenerating and reviewing the schemas,
updating fixtures, then deliberately changing the compatibility constant.

## Process and safety boundary

`AppServerJobDispatchAdapter` owns one `codex app-server --stdio --strict-config` child per bounded,
explicitly selected `provider: codex` job. It never invokes the terminal Codex adapter, Claude, or
another provider as a fallback. The process cwd is the validated absolute job cwd. `read-only` and
`workspace-write` are forwarded exactly; `danger-full-access`, `never` approvals, automatic review,
trust bypass, and permission-bypass flags are never produced. Approval policy is `on-request` and
the reviewer is `user`. An omitted model stays omitted. Opaque `role` is not interpreted.

The supervisor correlates request ids, accepts notifications between or before unrelated responses,
handles split frames, and bounds both individual frames and aggregate stdout/stderr. Malformed
JSON-RPC, unexpected response shapes, request timeout, output overflow, EOF, process errors, and
non-zero exits are explicit failures with captured bounded diagnostics. A job id may be submitted to
one adapter instance only once, and a settled guard suppresses duplicate terminal notifications.
Terminal cleanup clears timers, closes stdin, sends `SIGTERM`, rejects pending requests, releases a
held lease once, and drops process ownership.

Only structured App Server fields are control data. `turn/completed.status` decides completion;
`item/completed` agent-message text may become an untrusted human-readable summary but is never
executed or treated as a control instruction. Reported `thread/tokenUsage/updated.last` integer
counts map to neutral usage. Missing usage remains absent, never zero. Other notifications are
ignored or surfaced as neutral progress without inventing semantics. When the broker supplies its
existing artifact store, the validated terminal agent message is persisted as `codex-result.txt`,
content-addressed, attributed to the producing job, and returned by descriptor. Persistence failure
fails the job closed rather than emitting a completion whose result cannot be resolved. A standalone
adapter without an artifact store retains the original empty-artifact behavior.

Cancellation sends `turn/interrupt`; timeout sends the same interrupt and settles `timedOut` while
terminating the child. A confirmed interrupted turn after operator cancellation settles cancelled.
A malformed protocol is a terminal failed result. EOF, non-zero exit, and process error before a
validated terminal notification use `RUNTIME_INTERRUPTED`; the job control plane persists those as
`JobLifecycle.status = interrupted`, retains correlation diagnostics, and never redispatches.
Broker restart retains A3 behavior: stored queued/dispatched/running work becomes interrupted and is
not resumed. Live App Server execution remains fixture-proven and live-unverified until Gate 2.

## Lease identity, sharing, and fencing

The manager canonicalizes both repository and worktree with absolute resolution plus filesystem
`realpath`, including symlinks. Its exact target key is:

```text
<canonical-repository-path> NUL <canonical-worktree-path>
```

Ownership is exactly one `job:<uuid>` or `session:<uuid>`. Read-only leases may share one key only
with other read-only leases. A workspace-write lease conflicts with every active lease at that key,
and any active workspace-write lease blocks every new lease. Access comes only from the explicit
sandbox, never provider/model/role/output.

Every acquisition is serialized in-process, persisted and fsynced before it is returned, and gets a
strictly increasing fencing token for that canonical key. Renewal extends expiry without changing
the token. Validation requires the same lease id, owner, key, current highest token, unexpired held
state, and non-orphan status. Expiry is durably recorded. An expired owner fails validation after a
replacement acquires a larger token. A stale release is rejected; repeating release with the exact
current token is idempotent. App Server write jobs heartbeat at half the lease TTL and release once
on every controlled terminal path.

## Persistence, recovery, and manual remediation

Lease snapshots are schema-validated, uniquely event-tagged, append-only, and fsynced at:

```text
<state-directory>/control-plane/leases.jsonl
```

Replay keeps the latest snapshot per lease, rejects corruption, unsupported versions, blank middle
records, and duplicate event ids, and ignores only an unterminated crash tail. A held, unexpired
lease found at startup becomes a durable orphan because the previous process owner cannot be
verified. Orphans fail validation and block acquisition. Expired records become released.

Recovery never runs Git cleanup. `assessOrphanCleanup` always returns `safeToDelete: false`; with the
default inspector it assumes dirty because cleanliness is unproven. An injected repository inspector
may report status for evidence, but a dirty worktree is an explicit cleanup refusal. An operator must
inspect the job, repository, worktree, branch, and changes, preserve or commit user work as
appropriate, and then call explicit orphan resolution with the matching fencing token. Resolution
releases only the lease record; it never deletes a directory, branch, or Git worktree.
